# McNdroid тАФ CADE Budget Adaptation (Monthwise)

This repository contains the code for **McNdroid**, an Android malware detection system based on a **Contrastive Autoencoder (CADE)** with budget-constrained continual adaptation. The model is trained on features from 2013 and evaluated month-by-month across test years (2014тАУ2025), simulating a realistic deployment scenario where only a limited number of new samples can be labelled per month (the *budget*).

## Overview

The pipeline has three stages:

1. **Data preparation** (`mcndroid_monthwise_split_and_loader.py`) тАФ splits the master metadata CSV into one CSV per calendar month so both adaptation scripts can iterate month-by-month.
2. **CADE adaptation** (`mcndroid_cade_budget_adaptation_monthwise.py`) тАФ trains a Contrastive Autoencoder on 2013 data, then adapts it month-by-month with a limited labelling budget.
3. **Chen et al. (2023) adaptation** (`mcndroid_chen_budget_adaptation_monthwise.py`) тАФ implements the Hierarchical Contrastive Learning baseline from Chen et al. (USENIX Security 2023) on the same monthwise setup for comparison.

For each month the CADE script:
- Detects concept drift using MAD-based anomaly scoring in the latent space.
- Selects up to `budget` samples for adaptation (using one of three selection policies).
- Fine-tunes the CADE and retrains the classifier on the accumulated memory.
- Evaluates on the held-out samples for that month.

Results are reported across three temporal regimes тАФ **IID** (2014), **Near** (2016тАУ2017), and **Far** (2018тАУ2025).

## Step 1 тАФ Prepare Monthwise Data

> **This step must be completed before running the CADE script.** The CADE script reads the per-month CSV files produced here to know which APK hashes belong to each calendar month.

### What the data loader does

`mcndroid_monthwise_split_and_loader.py` reads the master metadata CSV (`final_hash_date_label_family.csv`) and:

1. Parses the `date` column and derives a `YYYY-MM` month string for every sample.
2. Writes one CSV per month under `monthwise_data/data/monthwise_csv/<year>/<YYYY-MM>.csv`.
3. Writes a `month_summary.csv` with per-month sample counts for a quick sanity check.

### Running the data loader

From inside the `Mcndroid/` directory:

```bash
python3 mcndroid_monthwise_split_and_loader.py
```

By default the script reads `final_hash_date_label_family.csv` from the current directory and writes output to `monthwise_csv/`. To use different paths, call the function directly (see below) or edit the `__main__` block at the bottom of the script.

To override paths inline:

```python
from mcndroid_monthwise_split_and_loader import split_csv_monthwise

summary = split_csv_monthwise(
    csv_path="path/to/final_hash_date_label_family.csv",
    out_dir="monthwise_data/data/monthwise_csv",
)
print(summary)
```

### Using the loader programmatically

The script exposes three helper functions you can import in your own notebooks or pipelines:

**`split_csv_monthwise(csv_path, out_dir)`** тАФ splits the master CSV into per-month files and returns a summary DataFrame.

```python
from mcndroid_monthwise_split_and_loader import split_csv_monthwise

summary = split_csv_monthwise(
    csv_path="final_hash_date_label_family.csv",
    out_dir="monthwise_data/data/monthwise_csv",
)
print(summary.head(12))
```

**`build_hash_to_month_map(csv_path)`** тАФ returns a `{sha256_hash: 'YYYY-MM'}` dictionary, useful for lookup without touching the per-month files.

```python
from mcndroid_monthwise_split_and_loader import build_hash_to_month_map

hash_to_month = build_hash_to_month_map("final_hash_date_label_family.csv")
print(hash_to_month["abc123..."])  # e.g. '2019-05'
```

**`split_loaded_arrays_monthwise(X, y, hashes, hash_to_month)`** тАФ splits already-loaded NumPy arrays `(X, y, hashes)` into a `{'YYYY-MM': (X_month, y_month, hashes_month)}` dictionary.

```python
from mcndroid_monthwise_split_and_loader import (
    build_hash_to_month_map,
    split_loaded_arrays_monthwise,
)

hash_to_month = build_hash_to_month_map("final_hash_date_label_family.csv")
month_buckets = split_loaded_arrays_monthwise(X, y, hashes, hash_to_month)

for month, (Xm, ym, hm) in month_buckets.items():
    print(month, Xm.shape, ym.sum())
```

**`flatten_yearwise_loader_to_months(yearly_loader_fn, years, split, csv_path)`** тАФ wraps an existing year-wise loader and returns a merged month-wise dictionary in one call.

```python
from mcndroid_monthwise_split_and_loader import flatten_yearwise_loader_to_months
from mcndroid_cade_budget_adaptation_monthwise import load_static

months = flatten_yearwise_loader_to_months(
    yearly_loader_fn=load_static,
    years=[2013, 2014, 2016, 2017, 2018],
    split="train",
    csv_path="final_hash_date_label_family.csv",
)
```

### Running with SLURM

If you have access to a SLURM cluster, the provided `graph.slurm` runs all 3 graph modality seeds as a job array (2 at a time). To run all four modalities, create one `.slurm` file per modality by changing the `--modality` flag and the job name. Example submission:

```bash
sbatch graph.slurm
sbatch static.slurm
sbatch dynamic.slurm
sbatch all.slurm
```

The array maps `SLURM_ARRAY_TASK_ID` тЖТ seeds `[42, 123, 999]` and run IDs `[run1, run2, run3]`.

> **Important:** Run the data loader (Step 1) before submitting any SLURM jobs. The loader only needs to be run once тАФ all four modality jobs read the same monthwise CSV files.

## Step 3 тАФ Run Chen et al. (2023) Adaptation

This step runs the **Chen et al. (USENIX Security 2023)** Hierarchical Contrastive Learning baseline on the same monthwise setup, using the same budgets and seeds as Step 2. It reads the same monthwise CSV files produced in Step 1, so **Step 1 must be completed first**.

The Chen method differs from CADE in three key ways:
- Uses a **Hierarchical Contrastive Loss** that distinguishes malware at the family level (same family тЖТ strongly pulled together, different family тЖТ weakly pulled together, benign/malware тЖТ pushed apart).
- Uses **Pseudo Loss** (uncertainty score based on k-nearest neighbour distances in latent space) instead of MAD-based anomaly scores for sample selection.
- Uses **warm-start retraining** тАФ the model continues from its previous weights each month rather than fine-tuning a separate encoder.

All training hyperparameters (epochs, learning rate, batch size, etc.) are fixed per the paper and do not need to be passed on the command line.

> **Tip:** If you don't have a GPU, replace `--device cuda:0` with `--device cpu`.

## Arguments Reference

### CADE script (`mcndroid_cade_budget_adaptation_monthwise.py`)

| Argument | Default | Description |
|---|---|---|
| `--modality` | *(required)* | Feature modality: `static`, `graph`, `dynamic`, or `all` |
| `--budgets` | `50,100,200,400` | Comma-separated list of labelling budgets to evaluate |
| `--device` | auto-detect | PyTorch device, e.g. `cuda:0` or `cpu` |
| `--seed` | `42` | Random seed for reproducibility |
| `--run_id` | `run1` | Label for this run, used in output filenames |
| `--min_epochs` | `150` | Minimum training epochs before early stopping is allowed |
| `--max_epochs` | `250` | Maximum training epochs |
| `--patience` | `10` | Early stopping patience (epochs without improvement) |
| `--min_delta` | `1e-4` | Minimum improvement to reset the early stopping counter |
| `--clf_epochs` | `15` | Epochs to train the latent classifier |
| `--finetune_epochs` | `3` | Epochs to fine-tune the CADE on each month's adaptation set |
| `--finetune_lr` | `1e-4` | Learning rate for fine-tuning |
| `--selection_policy` | `balanced_drift_topk` | Sample selection: `balanced_drift_topk`, `drift_topk`, or `random` |

### Chen script (`mcndroid_chen_budget_adaptation_monthwise.py`)

All model hyperparameters (encoder architecture, learning rate, batch size, epochs) are fixed per the paper and do not need to be passed on the command line.

| Argument | Default | Description |
|---|---|---|
| `--modality` | *(required)* | Feature modality: `static`, `graph`, `dynamic`, or `all` |
| `--budgets` | `50,100,200,400` | Comma-separated list of labelling budgets to evaluate |
| `--device` | auto-detect | PyTorch device, e.g. `cuda:0` or `cpu` |
| `--seed` | `42` | Random seed for reproducibility |
| `--run_id` | `run1` | Label for this run, used in output filenames |

### Fixed Chen hyperparameters (all modalities)

| Hyperparameter | Value | Description |
|---|---|---|
| Encoder hidden dims | `[512, 384, 256]` | FC layers before 128-dim latent |
| Latent dim | `128` | Embedding space size |
| Classifier hidden | `[100, 100]` | MLP head on top of encoder |
| Batch size | `1024` | Larger batches produce more contrastive pairs |
| Learning rate | `0.001` | Initial LR, decayed by 0.95 every 10 epochs |
| Initial epochs | `250` | Epochs for full training on 2013 data |
| Warm-start epochs | `100` | Epochs for monthly retraining |
| Warm LR ratio | `0.05` | LR for warm retraining = `lr ├Ч 0.05` |
| Pseudo loss k | `29` | Nearest neighbours for uncertainty scoring |

Each **CADE result JSON** contains:
- `per_month` тАФ per-month metrics: F1, FNR, FPR, ROC-AUC, drift percentage, and sample counts
- `iid` / `near` / `far` тАФ averaged metrics over each temporal regime
- Metadata: `modality`, `budget`, `seed`, `run_id`, `selection_policy`, `train_year`

Each **Chen result JSON** contains the same structure, plus:
- `method` тАФ always `"chen_al_2023"` for easy identification when comparing results

## Modalities

| Modality | Feature source | Autoencoder hidden dims | Latent dim | Scaler |
|---|---|---|---|---|
| `static` | Static analysis (permissions, API calls, etc.) тАФ sparse | `[1024, 512, 256]` | 64 | MaxAbsScaler |
| `graph` | Graph-based features (call graph embeddings) тАФ dense | `[1024, 512, 256]` | 64 | StandardScaler |
| `dynamic` | Dynamic analysis (runtime behaviour) тАФ sparse | `[2048, 1024, 512]` | 128 | MaxAbsScaler |
| `all` | Concatenation of static + graph + dynamic, hash-aligned | `[4096, 2048, 1024]` | 256 | Per-modality, then concat |
